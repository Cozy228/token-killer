# SCIP consumption = official TS binding + thin streaming importer (no pbjs binding, no hand-written decoder)

Status: accepted

`index.scip` is consumed via the **official Apache-2.0 TypeScript binding `@scip-code/scip`**
(generated by `protoc-gen-es`; v0.8.1, depends on `@bufbuild/protobuf` ^2.11.0, Apache-2.0
AND BSD-3-Clause), not by generating a parallel pbjs binding and not by hand-writing a full
protobuf decoder. Both of those are **rejected**: pbjs would duplicate an official binding,
and a hand-written nested-message wire decoder is error-prone on a low-frequency optional
path.

`@scip-code/scip` + `@bufbuild/protobuf` are **precisely version-locked build dependencies**,
bundled by tsdown into a **separate, lazy-loaded SCIP chunk**. After install there is **no npm
runtime dependency and no native addon**; the chunk loads **only when an `index.scip` is
detected**.

**Streaming, not whole-Index decode.** The official schema warns a full `Index` can use a lot
of memory, so the production path does **not** call `fromBinary(IndexSchema, wholeFile)`. tk
hand-writes **only the top-level Index framing** (read a field tag + its length-delimited
payload), then decodes each sub-message with the official `MetadataSchema` / `DocumentSchema`
/ `SymbolInformationSchema`. Each Document is decoded, written to SQLite staging, and
released. This mirrors the official Go `ParseStreaming` while hand-writing **no** nested SCIP
message wire decoder.

## Consequences

- Capability state: `ScipIndexer` = **Optional at runtime**; `ScipConsumer` +
  `StreamingScipImporter` = **Required**; `WholeIndexDecode` = tests / small fixtures only.
- **Range:** the current protocol prefers typed `single_line_range` / `multi_line_range`; the
  packed 3/4-int range is deprecated. The consumer is **typed-first, packed fallback**, and
  preserves each Document's **position encoding (UTF-8 / UTF-16 / UTF-32)** when normalizing
  char offsets. `symbol_roles` is parsed as a **bitmask** (`role & 0x1`), never enum equality.
- **Failure:** any decode/import failure rolls back the whole generation and transparently
  falls back to tree-sitter / the TypeScript checker (ties to the E index_generation counter
  and J's fail-open).
- The mapping from decoded Documents to tk edges still adapts graphify's two-pass method
  (PolyForm-NC, rewrite before distribution); only the *decode source* changed (official
  Document decode, streamed) — not the edge mapping.
- Supersedes the D.SCIP appendix's earlier "pbjs static module" recommendation (which also
  mis-stated runtime-zero-dep — pbjs static output imports `protobufjs/minimal`).
