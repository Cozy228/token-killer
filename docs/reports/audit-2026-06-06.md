# Comprehensive code audit — Token Killer (`tk`) — 2026-06-06

> Read-only audit across eight dimensions. See `docs/comprehensive-code-audit-goal.md` for the audit mandate.

## Summary

Token Killer is a well-structured codebase with clear architectural intent, strong fail-open guardrails,
and a consistent handler pattern across 55+ handlers. The hot path (cli → router → executor → pipeline)
is short and clean. **The single most impactful problem is that ADR 0001 (evidence-class capping) remains
largely unimplemented**: ~20 handlers still carry `CAP_*` constants and emit `+N more` / `... +N more`
markers that violate the "no fake-complete output" product invariant (PRINCIPLES.md). The Safe Compression
Gate (`base.ts:82-98`) is blind to tk's own omission markers — exactly the defect ADR 0001 diagnosed.
A close second is the hook protocol non-conformance (ADR 0005 §6): `toProtocol()` emits a non-standard
JSON shape no real host reads, making the shipped hook inert. Otherwise the codebase is clean across
most dimensions: the security posture is strong (inspect privacy holds, no ReDoS found, command injection
surfaces well-guarded), the test architecture is honest about its debt, and the overall design is coherent.

## Top findings (ranked by severity × confidence)

### 1. ADR 0001 capping removal not implemented — ~20 handlers still emit `+N more` markers `[CORRECTNESS-FAILOPEN]` severity:high confidence:confirmed

- **Where:** `src/handlers/python/pip.ts:98,111`, `ruff.ts:119`, `pytest.ts:158,204`, `js/next.ts:145`, `packageList.ts:120`, `prettier.ts:104`, `test.ts:174`, `playwright.ts:156`, `system/env.ts:160,195`, `json.ts:92,127`, `log.ts:89`, `pipe.ts:114`, `summary.ts:159,182`, `format.ts:134,206,320`, `read.ts:130`, `testRunner.ts:60,64`, `deps.ts:58`, `git/branch.ts:47`, `graphite.ts:52`, `git/compactDiff.ts:14-15,77,87`, `common/grepFilter.ts:236,241`, `listLike.ts:188`, `cloud/psql.ts:81,136`, `container.ts:8-10`
- **What:** ADR 0001 (accepted, docs/adr/) mandates removing all evidence-capping (`CAP_*` constants) and `+N more` markers, replacing them with a two-step ladder: (1) lossless reduction below budget, (2) complete-replacement summary + snapshot pointer above budget. Today ~20 handlers still carry the old RTK-style caps and emit `... +N more`, `… +N more`, `[N more lines]`.
- **Why it's a problem:** Direct violation of PRINCIPLES.md "no fake-complete output" invariant. The Safe Compression Gate (`base.ts:82-98`) regex patterns don't match tk's own markers (`... +N more`, `… +N more`), so capped output passes the gate as `qualityStatus: "passed"`. An agent receives incomplete context thinking it's complete.
- **Better path:** Implement the two-step ladder per handler family; remove `CAP_*` constants; set `omission: { kind, rawPointer }` on return. After all handlers done, retire the output-sniffing regex per ADR 0001 §5.
- **Evidence:** Grep confirms `... +N more` / `… +N more` across 20+ handlers. ADR 0001 §"Handler audit" lists every affected handler — none acted on.

### 2. Hook protocol non-conformance — `toProtocol()` emits shape no real host reads `[SECURITY]` severity:high confidence:confirmed

- **Where:** `src/hook/copilot.ts:26-32`
- **What:** `toProtocol()` emits `{ decision, rewritten_command, reason, additional_context }` — a tk-invented shape. VS Code expects `hookSpecificOutput.{permissionDecision, updatedInput, additionalContext}`. Copilot CLI expects `permissionDecision` + `modifiedArgs`. Neither reads tk's shape.
- **Why it's a problem:** ADR 0005 §6: "Until a host-protocol adapter is added, the shipped hook does nothing on a real host." The hook is a key delivery tier. If installed but inert, it cannot govern expensive direct reads/searches.
- **Better path:** Add per-dialect adapter: detect dialect from normalized event (already `ev.dialect`), emit VS Code shape vs Copilot CLI shape per ADR 0005 §Consequences item 1.
- **Evidence:** ADR 0005 decision 6 + code reading confirmed.

### 3. Eager import of all modules on hot path `[PERF]` severity:high confidence:confirmed

- **Where:** `src/cli.ts:2-24` (static imports)
- **What:** `cli.ts` statically imports 21 modules, including `handlers/index.ts` (all 55+ handlers), hook, inspect, context, shim, telemetry. `tk git status` pays the import cost for ~16k LOC.
- **Why it's a problem:** On the hot path, startup latency is load-bearing. Cold-path surfaces (inspect, context, hook, shim, telemetry, gain, config) are imported eagerly but only needed on their specific `ParseMode` branches.
- **Better path:** Dynamic `import()` for cold-path surfaces. Keep hot-path imports (router, executor, pipeline, handlers/index) static.
- **Evidence:** 8 of 21 imports are cold-path-only surfaces, all on `ParseMode` branches never touched on `tk <cmd>`.

### 4. Handler `truncate()` function duplicated 8 times `[DETOUR]` severity:medium confidence:confirmed

- **Where:** `src/handlers/python/pytest.ts:21`, `mypy.ts:20`, `js/next.ts:29`, `tsc.ts:25`, `dotnet/dotnet.ts:12`, `system/summary.ts:14`, `git/hostingCli.ts:83`, `graphite.ts:14`
- **What:** Eight handlers define identical `truncate(text, maxLen)` with `maxLen - 3` + `"..."` logic. Additionally, `readLike.ts` has its own level parsing instead of using `common/level.ts`.
- **Why it's a problem:** Maintenance hazard — a fix must be repeated 8 times. Violates "reuse `src/handlers/common/**`" principle.
- **Better path:** Extract one `truncate()` into `src/handlers/common/` or `src/core/text.ts`.
- **Evidence:** Grep confirms 8 identical copies.

### 5. `outputLimit.ts` is dead no-op code `[UNREASONABLE]` severity:medium confidence:confirmed

- **Where:** `src/core/outputLimit.ts:3-13`
- **What:** `limitLines()`, `limitChars()`, `limitOutput()` all `return text` unchanged. Called from `base.ts::makeFilteredResult()` but have no effect. `--max-lines`/`--max-chars` CLI flags parsed into `TkOptions` but never applied.
- **Why it's a problem:** CLI help advertises behavior that doesn't exist; dead code confuses readers.
- **Better path:** Implement or remove the functions + deprecate flags.

### 6. `base.ts` gate blind to tk's own omission markers `[CORRECTNESS-FAILOPEN]` severity:medium confidence:confirmed

- **Where:** `src/handlers/base.ts:82-98` (`outputOmitsContent()`)
- **What:** Regex only matches legacy phrasings (`+N more matches`, `[N more lines]`). Does NOT match `... +N more`, `… +N more`, `[... +N more]` — the exact forms tk handlers emit.
- **Why it's a problem:** Exact defect ADR 0001 §Context diagnosed: "the gate is blind to tk's own capping." Capped output passes with `qualityStatus: "passed"`.
- **Better path:** Short-term: add `... +N more` / `… +N more` patterns. Long-term per ADR 0001 §5: structured `omission` field, retire regex sniffing.
- **Evidence:** The `…` (Unicode ellipsis U+2026) used in `pytest.ts:204` is absent from patterns.

### 7. `test:ci` gate excludes migration debt — honest but incomplete `[TESTING]` severity:high confidence:confirmed

- **Where:** `package.json:30`, `vitest.migration.config.ts:14`
- **What:** `test:ci` runs only `test:product` (14 test files). The 63 `rtk*Behavior` tests are in migration suite, run report-only via `|| true`. A handler refactor that breaks `rtk*Behavior` but has no product test keeps the gate green.
- **Why it's a problem:** By design (migration debt documented), but means 63 handler behavior tests never block PRs.
- **Better path:** Move completed `rtk*Behavior` tests into product suite as migration finishes. Add `test:ci:migration-report` step (verbose, no `|| true`).
- **Evidence:** `vitest.migration.config.ts:11`: "This suite is expected to be red until migration completes."

### 8. No enforcing test for retention P0 `[TESTING]` severity:medium confidence:confirmed

- **Where:** `tests/` (absence), `tests/unit/handlers/fixtureContent.test.ts`
- **What:** `fixtureContent.test.ts` checks `critical` strings are present and `LOSSY_OMISSION_PATTERNS` are absent — but `LOSSY_OMISSION_PATTERNS` has the same blind spot as `outputOmitsContent()` (#6). No quantitative retention probe (e.g., "does output contain all 15 error locations from input?").
- **Why it's a problem:** PRINCIPLES.md: "P0 retention pass rate must be 100%" — but no test can assert this.
- **Better path:** Create probe-based tests per evidence class (diagnostics, test failures, search matches, SQL rows). Start with highest-risk handlers (pytest, grep, tsc, ruff).

### 9. `hook/install.ts` writes bare `tk hook copilot` command — fails on Windows `[SECURITY]` severity:medium confidence:confirmed

- **Where:** `src/hook/install.ts:38`
- **What:** `buildCopilotHookConfig()` hardcodes `command: "tk hook copilot"` — bare executable, PATH-dependent.
- **Why it's a problem:** ADR 0005 §5: on Windows PowerShell, bare `tk` fails with `CommandNotFoundException`. The probe worked only with absolute node path. Without fix, hooks installed on Windows are inert.
- **Better path:** Resolve absolute `node` + `cli.js` path at install time per ADR 0005.
- **Evidence:** `install.ts:38` + ADR 0005 §5 Corollary 2.

### 10. `readLike.ts` `readInternally()` reads entire files eagerly into memory `[PERF]` severity:medium confidence:confirmed

- **Where:** `src/handlers/common/readLike.ts:132-185`
- **What:** Reads all files into `string[]` via `readFile()`, concatenates, then `split(/\r?\n/)`. For a 10MB file: ~30MB allocated (buffer + string + split array).
- **Why it's a problem:** `tk read` targets large files; the aggressive level is designed to avoid exactly this cost.
- **Better path:** Stream line-by-line with `readline`; only buffer output, not input.

### 11. `compactDiff.ts` LCS allocates O(n×m) DP table `[ALGO]` severity:medium confidence:confirmed

- **Where:** `src/handlers/common/diff.ts:22-25` (`lcsChanges()`)
- **What:** Allocates `(oldLines+1)×(newLines+1)` dense array. Two 10K-line files = 100M elements before diffing starts.
- **Why it's a problem:** Hot-path (`tk diff`); no size guard, no fallback to real `diff`.
- **Better path:** Hirschberg (O(n+m) space) or Myers (O((n+m)×D) where D is edit distance) for line-level diffing.

### 12. Shared-state `rawStore.ts` writes without atomicity `[CORRECTNESS-FAILOPEN]` severity:medium confidence:confirmed

- **Where:** `src/core/rawStore.ts:41` (single `writeFile`), plus `history.ts:69`, `governance.ts:51` (append)
- **What:** `rawStore` writes multi-line content (header + stdout + stderr) in one `writeFile` — for large output this exceeds `PIPE_BUF` and is torn-write-prone. History/governance JSONL appends are likely atomic (sub-4KB).
- **Better path:** Write to temp file, then atomic rename for `rawStore`.
- **Evidence:** `rawStore.ts:31-42` constructs multi-line string, writes in single `writeFile`.

### 13. `log.ts` `renderSection()` uses O(n²) lookup `[ALGO]` severity:low confidence:confirmed

- **Where:** `src/handlers/system/log.ts:78-79`
- **What:** `bucket.unique.find(line => normalizeLogLine(line) === normalized)` — re-runs regex-based normalization per comparison.
- **Better path:** Store map from normalized → original during initial pass. O(1) lookup.

### 14. Double line-splitting per compression `[ALGO]` severity:low confidence:confirmed

- **Where:** Handlers split by `/\r?\n/` for processing, then `base.ts:83` splits again for quality gate.
- **What:** Each command output split into lines at least twice.
- **Better path:** Pass pre-split `lines[]` alongside raw text to `outputOmitsContent()`.

### 15. `any` types proliferate in JSON parsing handlers `[HYGIENE-DOCS]` severity:medium confidence:confirmed

- **Where:** `js/packageList.ts:30,36` (`pkg: any`), `python/pip.ts:27` (`payload: any[]`), `cloud/aws.ts`, `dotnet/dotnet.ts`
- **What:** Several handlers use `any` for `JSON.parse()` results while others (ruff, pip partially) define typed interfaces.
- **Better path:** Define typed interfaces; use `unknown` + type guards.

### 16. Stale hardcoded handler lists in CI scripts `[HYGIENE-DOCS]` severity:medium confidence:confirmed

- **Where:** `scripts/validate-docs.sh:13-25` (`PROGRAMS` array), `scripts/check-test-presence.sh:81` (`KNOWN_HANDLERS`)
- **What:** Hardcoded lists drift from actual handler registry (55+ entries). `check-test-presence.sh` dynamic check is more robust.
- **Better path:** Derive lists dynamically from `handlers.map(h => h.name)`.

### 17. `readLike.ts` own level parser diverges from `common/level.ts` `[DETOUR]` severity:medium confidence:confirmed

- **Where:** `src/handlers/common/readLike.ts:59-121` vs `common/level.ts`
- **What:** `readLike.ts` defines its own `ReadLevel`, `parseReadLevel()`, flag loop. `searchLikeHandler` and `format.ts` use `common/level.ts` correctly.
- **Why it's a problem:** Two divergent level-parsing implementations. `readLike.ts` accepts `"balance"` → `"balanced"` (one-off alias).
- **Better path:** Use `parseLevel()` from `common/level.ts`.

## By module

### core
Clean with three exceptions: `outputLimit.ts` dead stubs, `rawStore.ts` torn-write risk, `tokens.ts` consolidation done correctly (ADR 0004 §8). `pipeline.ts` is clean and minimal. `ansi.ts` correctly delegates to `strip-ansi`. `patterns.ts` single regex is safe.

### handlers
**Major finding:** ADR 0001 capping removal not implemented — ~20 handlers still emit `+N more`. **Duplication:** `truncate()` 8×, `readLike.ts` own level parser, two diff-compaction paths (`diff.ts` + `compactDiff.ts`). **Clean families:** `git/*` well-structured with RTK-parity comments; `js/tsc.ts`, `js/eslint.ts` clean; `cloud/curl.ts` proper truncation + recovery; `iac/terraform.ts` clean; `npx.ts` correctly re-dispatches.

### hook
**Critical:** Protocol non-conformance (ADR 0005 §6). **Clean:** `normalize.ts` thorough fail-open; `rewrite.ts` proper safety guards (redirect, pipe-into-xargs, mutating ops, interactive denylist). **Install:** bare `tk hook copilot` command.

### inspect
**Clean:** Privacy promise holds — `sources.ts` only discovers paths, never reads content. `unified.ts` constructs findings from aggregates. No raw content leakage. ADR 0003 scope-aware inspect implemented.

### context
**Not deeply audited.** `analyzer.ts`, `discover.ts`, `metrics.ts`, `rules/**` exist. Scope persisted per ADR 0003. Surface-level scan no issues.

### shim
**Clean:** `gate.ts` pure testable function. `interactive.ts` focused denylist. `path.ts` thorough recursion guard (strip + sentinel). `detect.ts` reasonable auto-detection. `programs.ts` interpreter denylist good safety net.

### telemetry
**Clean:** `build.ts` physically constructs only allow-listed fields — no `HistoryRecord` spread. Privacy holds. ADR 0004 consent model (two flags) implemented.

### entry (cli, router, parse, executor, types)
**Clean:** `cli.ts` fail-open correct (try/catch → failOpenPassthrough). `router.ts` exposes `routeSpecific` (ADR 0002). `executor.ts` proper Windows support (PATHEXT, batch-script, UTF-8 env). `types.ts` well-typed.

## Suspected (lower confidence)

1. **`container.ts` format functions (dockerPs, dockerImages, etc.) are monolithic** — ~500 lines of formatting; could be per-resource modules. Suspected maintainability issue. Confirm by measuring effort of adding a new docker subcommand.

2. **`searchLike.ts:143` and `compactDiff.ts:87` recovery hints use `tk --raw` (re-run) instead of snapshot path.** ADR 0001 §6 mandates snapshot recovery, not re-run. Suspected ADR non-compliance — confirm by checking all recovery hints.

3. **`context/**` modules (~2.6k LOC) not deeply audited** — `analyzer.ts`, `rules/**`, `optimizeCli.ts`. Surface scan no issues, but deep audit might find rule-specific problems.

4. **`history.ts` `listProjectHistories()` / `listProjectHistoriesSync()` near-duplicate** — async vs sync differ only in `readFile`/`readdir` vs sync variants. Suspected DETOUR.
