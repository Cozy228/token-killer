# Token Killer（tk）领导汇报速记

> 用途：不是逐字读，而是记住每个**关键词**，能接住领导任何追问。
> 一句话定位：**本地、确定性的命令代理 —— 在不丢证据的前提下，砍掉 AI 编程助手的噪声输出，省 token、省钱。**

---

## 0. 30 秒电梯陈述（背这段）

> AI 编程助手（Copilot / Claude Code）每次跑命令，都会把 `git status`、`npm test`、日志这些**几千行噪声**塞回模型上下文，token 费用和上下文窗口被无意识地浪费。Token Killer 在命令和模型之间放一个**本地代理 `tk`**，跑真实命令、**无损压缩**输出、记录省下的 token。**更进一步**：它还能把**企业级的 token 优化最佳实践（哪些上下文该精简、哪些设置该改）一键诊断并分发到每个开发者**，让省钱从"个人技巧"变成"全员标准"。核心原则是**"宁可不压，也不丢证据"**，所有数据**留在本地**，对企业是**纯降本、零数据外泄**。

**关键词**：本地代理 · 无损压缩 · 不丢证据 · **最佳实践一键分发** · 降本 · 数据不出本机

### 两条价值主线（这是骨架，先记住）

1. **运行时省 token（压缩）** —— 单机即时见效，命令输出无损压缩。
2. **治理面分发最佳实践（诊断 + 分发）** —— 体检每个人的上下文/配置浪费，把优化方案标准化、可复制地铺到全员。

> 第一条是"省"，第二条是"管"。领导真正买单的往往是第二条：**规模化、可复制、可统计**。

---

## 1. 痛点 → 解决的问题

| 痛点 | 关键词 |
|---|---|
| AI 助手把命令的全部输出回灌上下文，token 暴涨 | **无意识 token 膨胀** |
| 现有方案（如 RTK）粗暴 `truncate`，截断了报错位置/diff，误导模型 | **截断即误导** |
| 省了 token 却让模型拿错信息，比不省更糟 | **错误压缩比零压缩更糟** |
| 团队看不到到底省了多少、值不值 | **价值不可见** |

**我们解决的核心问题**：在**省 token** 和**不丢关键证据**之间，做一个**可证明、可恢复**的平衡。

---

## 2. 亮点（差异化，重点讲这 5 个）

1. **保留优先（Retention-first）**：先保证证据完整，再谈压缩。`file:line`、diff、报错位置**永不截断**（"位置类证据"）。
   - 关键词：**保留优先 · 位置类证据永不截断**
2. **诚实压缩，拒绝"假完整"**：禁用 RTK 式的 `+N more` 标记（假装列全了其实砍了）。超预算只走**两级阶梯**：先无损去重，再换成**完整聚合摘要 + 快照指针**。
   - 关键词：**禁止 +N more · 超预算阶梯 · 完整替换式摘要**
3. **命令感知（Command-aware）**：不是通用摘要，每种命令有专属处理器（git/test/grep/tsc…），按结构压缩。
   - 关键词：**按命令定制 · 30+ 处理器**
4. **可恢复 + 可度量的诚实账本**：压缩同时落地原始快照，输出里写明"怎么读回来"。价值用**四本独立账本**展示，**绝不相加**（实测数 ≠ 估算数）。
   - 关键词：**恢复契约 · 四账本不相加 · 诚实是护城河**
5. **确定性、本地、测试先行**：纯本地执行，无网络依赖，**零运行时依赖**，140 个测试。
   - 关键词：**确定性 · 零依赖 · 140 测试**

---

## 2.5 第二条价值主线：把最佳实践一键分发到全员（治理/分发面）★

> 这是上一版漏讲、但对企业最有分量的部分。压缩是"单机省"，这一块是"全员管"。

**它解决什么**：每个开发者的 AI 配置（指令文件、skill、agent、VS Code 设置）都在**无意识地往上下文里塞冗余**，而且各写各的、没人统一。Token Killer 把"省 token 的最佳实践"**变成可诊断、可分发、可统计的标准动作**。

**四步闭环（背这个流程）**：

1. **诊断（inspect）** —— `tk inspect` 只读体检，扫两类证据：运行时会话历史 + 静态上下文文件（指令/prompt/agent/skill）。找出"哪段上下文常驻浪费、哪条规则该下沉、哪个设置该改"。→ `src/inspect/`
2. **优化（optimize）** —— `tk optimize` 消费体检结果，把每条规则**下沉到最窄的、仍能生效的位置**（常驻指令 → 按路径 → 按需调用），并产出**安全机械修复**或**建议 diff**。→ `src/context/` + `src/context/rules/`
3. **分发（init + agentsmd + 注入）** —— `tk init` 一键把代理接到每个人的环境（按宿主选最高可用投递级别）；`tk optimize --token-budget-block` 幂等地把统一指令块写进用户级文件；注入块用守卫注释包裹、可干净卸载。→ `src/shim/init.ts`、`src/context/agentsmd.ts`、`src/shim/injection.ts`
4. **统计（telemetry）** —— 匿名设备哈希让企业能**统计装机量与留存**（多少人真的用上了、留下来了），衡量推广效果。默认全关、opt-in。→ `src/telemetry/`

**为什么领导爱听**：
- **可复制**：一条命令铺到全员，不靠每个人自己学技巧。
- **标准化**：上下文规则下沉、设置统一，把"个人经验"沉淀成"团队资产"。
- **可度量**：装机/留存 + 四账本，推广有数据交代。
- **不侵入**：默认只出建议 diff、写用户级目录，不强改任何人的项目仓库。

**关键词**：诊断→优化→分发→统计 四步闭环 · inspect 体检 · 规则下沉到最窄表面 · init 一键铺全员 · 装机留存可统计 · 不强改仓库

## 3. 效果 / 数字（能被追问的硬数据）

- 压缩率因命令而异：`rg` 从 0% 提升到 **88.5%**，整体在 dev 操作上 **60–90%** 区间。
- 账本里只展示**实测**的 `raw − delivered`（叫 `saved_tokens`），**不夸大、不估算混入**。
- 代码量 ~20k 行 TS，140 个测试，零运行时依赖。

**关键词**：rg 0→88.5% · 实测 saved_tokens · 不混估算

> ⚠️ 被追问"到底省多少"时的标准答案：**"看命令类型，噪声越大省越多；我们只报实测，不报估算。"** 别承诺单一百分比。

## 4. 安全 & 合规（领导最关心，必背）

| 维度 | 事实 | 关键词 |
|---|---|---|
| 数据位置 | 一切写到 **`~/.token-killer/`** 用户级目录，**从不写项目仓库** | **数据不出本机** |
| 隐私 | 原始证据（命令/路径/会话）默认**不输出、不进遥测**，仅按次显式 opt-in | **默认不留痕** |
| 遥测 | **默认全关**，分两个独立开关（本地导出 / 网络上传），互不蕴含；网络上传还需内置 endpoint | **双开关 · 默认 off** |
| 遥测身份 | 只发**匿名设备哈希**（随机盐的 SHA-256），无主机名/用户名/路径，删除本地状态文件即可清除 | **匿名 · 可清除** |
| 失败策略 | 任何内部解析/配置错误一律 **fail-open → 放行原始命令**，绝不崩溃、绝不挡住开发 | **失败即放行** |
| 交互命令 | 编辑器/登录/分页等交互命令**直接透传**，不捕获 | **TTY 透传** |

> 一句话总结安全：**"本地优先、默认不上报、匿名可清除、出错只放行不阻断。"**

## 4.6 Windows 11 适配（公司主力机型，必看）

> 公司绝大多数是 Win11，这一节单独讲清楚，领导一定会问。

**结论：公司主场景（VS Code + Copilot CLI on Win11）已实现，并在 Win11 真机上跑通验证。**

| 能力 | Windows 上怎么做 | 状态 |
|---|---|---|
| 命令接管 | 在用户目录放一组同名"包装器"，系统按 Windows 规则自动优先识别 | ✅ 真机验证 |
| VS Code 接入 | 改 VS Code **用户级设置**（不是系统 PATH），把代理放到终端环境最前 | ✅ 真机验证 |
| Copilot CLI 接入 | 写用户目录下的接线配置 | ✅ 真机验证 |
| 中文不乱码 | 适配 Windows 中文控制台编码 | ✅ 已实现 |

- **不需要管理员、不改系统 PATH**：Windows 上走 VS Code 自己的用户级终端设置，**正好符合公司"只能改用户级 PATH"的限制**。
- **唯一没覆盖**：在 VS Code / Copilot 之外、单独开一个 PowerShell 窗口手敲命令的情况——但那不是 AI 省 token 的目标场景。

**关键词**：Win11 真机验证 · 用户级 / 不改系统 PATH · 不要管理员 · 中文不乱码 · 主场景全覆盖

## 5. 关键入口（领导问"这块怎么做的"，心里有数）

> 这是给你自己的速查：每个大功能对应哪条命令、代码在哪。汇报时不用念，被追问能定位即可。

| 功能 | 一句话 | 命令 | 代码位置 |
|---|---|---|---|
| **压缩** | 命中专属处理器才压，否则原样透传；30+ 处理器按生态分类 | `tk <命令>` | `src/handlers/` |
| **三级投递** | 让助手用上 tk：Hook > Shim > 指令注入，按可用性自动选最高级 | — | `src/hook/`、`src/shim/` |
| **一键接线** | 自动识别环境、装好对应投递方式，可预演、可卸载 | `tk init` | `src/shim/init.ts` |
| **体检** | 只读扫描，找出上下文/配置里的浪费，给优化建议 | `tk inspect` | `src/inspect/` |
| **优化** | 消费体检结果，做安全修复或给建议 diff，默认不改项目文件 | `tk optimize` | `src/context/` |
| **遥测** | 匿名统计装机/留存，默认全关、两道独立开关 | `tk telemetry` | `src/telemetry/` |
| **账本/报表** | 四本独立账本，只报实测、绝不相加 | `tk gain` / `tk gain report` | `src/core/` |

**几个能撑场面的设计点（被追问时说）**：
- 三级投递严格按"高可用优先"：能用 Hook 用 Hook，不行退 Shim，再不行才退指令注入（靠模型自觉、概率性）。
- 体检**只诊断不强制**，默认看全局（user 级）配置，加 `--project` 才看当前仓库。
- 优化器**不是第二个扫描器**，它只消费体检结果，且默认只出建议、不动项目文件。
- 四本账本**永不相加**：实测省下的 token 绝不和估算值混在一起报。

## 6. 能优化哪些点 / 哪些可配置（领导会追这些）

### VS Code 设置优化
- **只有一项会被自动改且可一键还原**：开启 VS Code 自带的终端输出压缩开关（改前自动备份）。
- 其余有风险的设置**只给建议、不自动改**，例如：自动加载过多指令文件、给模型开放额外读取目录、自动发现 MCP 工具、单轮请求数过高（>15 偏费，控费档建议 8–12）等。
- **关键词**：只自动改一项 · 改前备份可还原 · 其余只建议

### Skills / Agents / 指令文件的裁剪
- 把每条规则**下沉到"最窄但仍能生效"的位置**：常驻指令 → 按路径加载 → 按需调用。能少进上下文就少进。
- 自动识别重复、冲突的规则，并按"加载频率类别"（常驻 / 按路径 / 按需）给出精简建议——**只算类别、绝不拿频率去放大估算数**。
- **关键词**：规则下沉到最窄处 · 去重去冲突 · 频率是类别不是乘数

### 注入给助手的提示语，是否可配置？
- 技术上**可以自定义**（接口预留了传入自定义文案的能力）；目前默认用内置文案，还没开放成用户配置项。
- 注入的内容用专属标记包裹，**幂等、可干净卸载**，不会污染文件。
- **诚实口径**：**"文案技术上可定制，目前用内置默认值，未开放为配置。"**

### 压缩力度可调
- 三档：精简 / 平衡 / 激进；也可手动限制最大行数、字符数；`--raw` 可随时关掉压缩看原始输出。
- **关键词**：三档力度 · 可限行数/字数 · --raw 兜底

## 7. 与 RTK 的关系（如果有人知道 RTK，会问）

- RTK = 同类先行者（Rust），我们对标并迁移其能力，但**不盲目继承它的截断**。
- 核心分野：RTK 把一切叫 "truncation"（截断），我们区分**无损去噪 vs 证据截断**，后者禁用。
- 我们独有：**Shim 投递层**、**四账本诚实度量**、**inspect 静态上下文分析**（RTK 都没有）。
- **关键词**：对标 RTK 但拒绝盲目截断 · Shim/四账本/静态分析是我们独有

## 8. 现状 & 风险（主动说，显得靠谱）

- 已落地：命令代理 + 30+ 处理器、三级投递、inspect + optimize、四账本、遥测、Windows GBK 编码兼容。
- 诚实风险点：
  - **指令注入靠模型自觉，不是 100% 触发**，所以才有 Hook、Shim 两个更确定的兜底方式。
  - **VS Code 里压缩走 Shim**（VS Code 本身的接口不支持改写命令的最终输出，这是平台限制，不是我们的缺陷）。
  - 部分语言（Go/Rust/Ruby）的处理器仍在完善中。
- **关键词**：注入非必中 → 有确定性兜底 · VS Code 压缩走 Shim · 处理器持续扩

### 重点说清楚 #1（VS Code Hooks 被组织禁用）

- **影响**：没有 Hook，VS Code 上**仍可用**——压缩走 **Shim**（PATH 包装器，策略无关、不依赖 Preview），指令注入兜底。但**拿不到** Hook 独有的能力：PreToolUse 治理（拦截昂贵的全仓搜索/超大文件读）、PostToolUse 同轮恢复提示、透明命令改写。
- **要 IT 做的事**：把组织策略里的 **`chat.enableHooks`** 从锁死 disabled 改为允许（它目前还是 VS Code Preview 特性，所以即便解锁也要接受"契约可能变"的前提）。
- **诚实口径**：**"Hook 不是必须项——没有它产品照样跑（走 Shim）；解锁它能解锁治理和透明改写这层更强的能力。"** 这样既提了需求，又不把方案绑死在一个权限上。

**关键词**：Hook 非必须 · 没有就走 Shim 兜底 · 解锁 chat.enableHooks 拿满血治理 · Preview 风险

## 附：可能被问到的刁钻问题 + 标准答案

- **"会不会改坏我的项目文件？"** → 不会。一切写用户级目录，optimizer 默认只出建议 diff，VS Code 设置写前备份可还原。
- **"压坏了怎么办？"** → 原始快照同turn落地 + 输出写明读回路径（恢复契约）；出错一律 fail-open 放行原始命令。
- **"数据会上传吗？"** → 默认全关，需两个独立 opt-in，上传只发匿名设备哈希，可一键 purge。
- **"省多少？"** → 看命令噪声量，60–90%，只报实测不报估算。
- **"和直接 truncate 有啥区别？"** → 我们禁用假完整（+N more），位置类证据永不截断，错误压缩比零压缩更糟是我们的第 8 条原则。
- **"这不就是个压缩工具吗？价值有限？"** → 压缩只是第一条主线；第二条是**把 token 优化最佳实践一键诊断并分发到全员**（inspect→optimize→init→telemetry），把个人技巧变成团队标准、还能统计装机留存。
- **"VS Code 里 Hook 被组织禁用了，还能用吗？"** → 能。Hook 不是必须项——压缩走 Shim（策略无关），照常省 token；解锁 `chat.enableHooks` 只是额外拿到治理拦截和透明改写这层更强能力。
