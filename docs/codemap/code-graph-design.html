Product Design

One base, two faces:
code graph for humans & agents

A product, not a release phase. One shared knowledge base — fused from code AST, git/commit history, tests, Confluence docs and Jira stories — projected to two faces: a human face (a generated Code Wiki + graph explorer for understanding and collaboration) and an agent face (a codegraph MCP that locates code and cuts token usage). tk owns no LLM and spends zero API tokens: the deterministic structure is tk's; all semantics are produced by the user's own AI agent and fed back in.

§1b

Input layer — multi-source ingestion

The base is fused from several sources, not just code. The dividing line is what can be extracted deterministically (code structure, git history, test wiring, commit→ticket regex — tk builds these locally, zero LLM) versus what needs semantic mapping (doc↔code, story↔code intent, business domain — these are agent-fed, tagged source=host_llm). Every node and edge carries a source tag and a confidence so the two are never confused.

Two ingestion modes for Confluence / Jira

1. Direct connector. tk pulls text via REST (base URL + token in config/env). The fetch is deterministic, but the doc↔code mapping still needs the agent. Enterprise-gated; opt-in.

2. Host-agent-fed (preferred). The user's agent — which already has Confluence/Jira/MCP access — reads them, computes the doc↔code / story↔code mapping, and writes structured nodes/edges into tk via the annotate API. Keeps tk LLM-free and puts the hard semantic mapping where an LLM belongs.

Why this fits "tk owns no LLM"

tk deterministically ingests everything a parser or git gives for free. The expensive, judgement-heavy parts — what a doc means for which code, which story a change implements, what business domain a module serves, and the Code Wiki prose — are what the host agent is good at and a parser is not. The agent (the user already pays for it) does that analysis and feeds it back, tagged source=host_llm with citations. tk stores, validates anchors, tracks staleness — and never holds an API key.

Provenance is the contract. Deterministic facts (source ∈ {static, git, test}) are trusted and reproducible. Agent-asserted facts (source ∈ {host_llm, confluence, jira}) carry confidence + required citations and are visibly marked as such in every projection — so a low-confidence doc↔code guess is never presented as a parser-proven call edge.
§5

Human face — Code Wiki + graph explorer

The human face turns the shared graph into something a person reads to understand the system and collaborate. It has two complementary products — a generated Code Wiki and an interactive graph explorer — both served from the same retrieval core, both LLM-free on tk's side (prose comes from the host agent, source=host_llm).

Code Wiki — generated documentation

Holistic, hierarchical, repository-level docs — the capability CodeWiki/RepoDoc/OpenDeepWiki demonstrate, rebuilt on tk's deterministic base:

  • Structure is graph-derived (deterministic): page tree from files/contains; module clustering; leaf-first ordering — like CodeWiki's hierarchical decomposition.
  • Prose is host-agent-fed (source=host_llm): the agent writes section text via tk_annotate — CodeWiki's "subscription mode" idea, except the user's own CLI agent does it, so tk pays no tokens.
  • Diagrams are graph-derived + validated + fallback (RepoDoc): mermaid rendered from real edges, syntax-checked, deterministic fallback if invalid — never LLM-hallucinated.
  • Every page cites sources (OpenDeepWiki SourceFiles): blocks carry file:line anchors + freshness; manual blocks never overwritten; stale blocks shown as stale.
  • Self-contained HTML export (CodeWiki --github-pages / GitNexus viewer): inline JSON into a template, no CDN, shareable as a CI artifact.
Graph explorer — navigation

The interactive view of the same graph — the UX Understand-Anything sets the bar for:

  • overview → lazy-expand → focus: PageRank "key files" + module clusters; expand children on demand via contains/imports; focus = 1-hop over calls.
  • Guided tours: an ordered walk over PageRank / entry-point nodes — derivable deterministically, no LLM.
  • Detail panels: signature + clickable file:line (vscode://file/… in the IDE) + a small graph-derived call snippet, plus any agent-fed summary/domain tags.
  • Same anchors as tk_read: human and agent point at identical evidence.
Delivery & cost. tk graph serve exposes both products over localhost HTTP (open in VS Code's Simple Browser); tk graph export --html emits a self-contained snapshot. Measurement: the human face reports navigation facts (nodes shown vs total, expand depth, anchors opened) — it does not claim token savings; its value is comprehension. The token story stays on the agent face.

MCP tool I/O contracts

Every response carries model-facing output, resolvable anchors, confidence, source provenance, stale/hash state, and measurement facts (returnedAnchors, returnedChars).

// tk_map — orient; cheapest rung of the ladder in { cwd, query?, token_budget? } out { answers:[{ file, line, signature, why, source, confidence }], clusters:[{ name, files[] }], next_read?, measurement } // tk_read — exact slice; edit_window for editing contexts in { cwd, target:{ symbol | file:range | node_id }, mode:"default"|"edit_window", token_budget? } out { content, anchors:[{file,line}], content_hash, confidence, candidates?[], // narrow hand-back when target is ambiguous measurement } // tk_search — text / symbol / callers / impact-lite in { cwd, query, mode:"text"|"symbol"|"callers"|"impact-lite", token_budget? } out { hits:[{ file, line, snippet, confidence, source }], impact?:{ callers[], co_changes_with[], owns[] }, // T1 facts, labelled measurement } // tk_verify — local diff + test-failure projection (never claims correctness) in { cwd, diff?, test_output?, token_budget? } out { changed_files[], changed_symbols[], failing_tests[], anchors[], raw_pointer?, measurement }

A fifth tool, tk_annotate, is exposed only in ingest mode (see §2b) — the host agent writes semantic / cross-source annotations and Code-Wiki prose back into the graph. It is not part of the resident read surface.

§2b

Semantic & cross-source ingestion protocol

This is the seam that lets tk stay LLM-free while still carrying semantic/domain/business knowledge, Confluence/Jira links, and Code-Wiki prose. The host agent (or the optional connector) writes through one narrow, validated API — tk graph annotate (CLI) / tk_annotate (MCP, ingest mode only). Annotations are additive: they attach to existing deterministic nodes, never mutate parser facts, and always carry source + confidence + citations.

// tk_annotate — host agent writes semantics / doc-links / wiki prose back in in { annotations: [ { target: { kind, qualified_name | "file:line-range" | node_id }, // must resolve kind: "summary"|"domain"|"tags"|"layer" | "doc_link"|"story_link"|"test_link"|"wiki_block", value: "…text / enum / string[] / markdown…", source: "host_llm"|"confluence"|"jira", citations: [ "src/auth/login.ts:42", "confluence://SPACE/page/123", "PROJ-456" ], // REQUIRED confidence: "high"|"medium"|"low", content_hash: "<hash of target at annotation time>" // for staleness } ] } out { accepted[], rejected:[{ target, reason }], stale_now[] }
Validation & safety rules
  • Every annotation must cite a resolvable source (file:line, Confluence page, Jira key); uncited → rejected.
  • The target anchor must resolve to the current graph or it is rejected (no dangling semantics).
  • Stored with source + confidence; never overwrites a deterministic fact or a human-authored block.
  • Goes stale automatically when the target's content_hash changes — surfaced as stale, not silently trusted.
  • Idempotent by (target, kind, source): re-annotating updates in place.
Lifecycle
# 1. tk builds the deterministic base (no LLM) tk graph index # 2. host agent enriches (its own tokens, not tk's) agent: read graph via tk_map/tk_read agent: map docs/stories ↔ code, infer domain, write Code-Wiki prose agent: tk_annotate({ … source:"host_llm" … }) # 3. both faces read the combined graph human: tk graph serve # Code Wiki + explorer agent: tk_search / tk_map # cut tokens # 4. code changes → annotations go stale → re-enrich
§4

Data model & static engine

One typed node table + one typed edge table (not a table per kind), with migrations from the first slice. The generic nodes(kind, …, source, confidence, content_hash, metadata_json) + edges(kind, …, source, confidence) schema absorbs every input source as new kind + source values — no new tables. v1 avoids LSP (smaller support matrix); ConceptNode is dropped — that role is now the agent-fed domain/summary annotation.

§12

Measurement harness: measure before the feature

The core question: how do we honestly evaluate the agent face's search/read token saving — in tk's "measured ≠ estimate" ledger model — before building the graph (ADR 0016)? (The human face is measured by comprehension, not tokens.)

WinWhat it isWithin-call measurable?Class
W1 output projectionthe graph's own response collapses container bodies into signaturesyesmeasured, like an existing handler
W2 loop avoidancethe agent skipped 5×grep + 3×read because one tk_* answeredno (counterfactual)online opportunity / offline A/B only
Headline benchmark figures (e.g. "47% fewer tokens") are overwhelmingly W2. So the harness's center of gravity must be trajectory-level A/B, not within-call compression ratio. tk's saved_tokens = raw − delivered (a within-call diff) does not transfer to a graph query — which is why ADR 0016 forbids the graph writing saved_tokens.

Track 1 — offline A/B eval

the measured proof · trajectory level
  • Fixed task set (11 categories), N=5 runs per arm per task, take the median (variance up to 30×).
  • BASELINE = graph MCP OFF; TREATMENT = graph MCP ON; both keep the host's built-in read/grep.
  • Primary metric = uncached_input_tokens delta, not total-incl-cached (cache-read >97% of usage).
  • Co-measure quality: localization F1 · AST validity · FAIL_TO_PASS ∧ PASS_TO_PASS.
  • fallback-replay → omission_bug_rate as the safety counterweight.

Track 2 — online opportunity accounting

live facts · estimate_kind=opportunity · never summed
  • Per query, record mechanical facts: nodes_returned · files_returned · tokens_returned.
  • tool_calls_collapsed = 1 (the single call standing in for the loop).
  • reads_avoided_in_session — cross-referenced with the trajectory analyser.
  • Only savings % shown is the Track-1 reference rate, labelled measured-in-eval, not on your machine.
  • Independently validated by Serena, which refuses token benchmarking and reports call counts / payload sizes instead.

Appendix A — how the surveyed projects measure token saving

Four measurement philosophies. Only A (trajectory A/B delta) measures W2; B/C answer a different question; D is unsubstantiated.

Bucket A — trajectory A/B delta (the imitable template)

§8

Open decisions — needs your call

The uncertain areas, made explicit. Each has a recommended default so the plan can proceed; flag any you want to change.

D1Confluence / Jira ingestion mode
Direct REST connector (tk holds a token) vs host-agent-fed (the agent reads them, writes via tk_annotate) vs both.
Rec: agent-fed first — keeps tk LLM-free and credential-free; add the connector as opt-in once the seam is proven.
D2Code Wiki prose — who writes it
Host agent via tk_annotate (subscription-mode, tk pays nothing) vs an optional tk-side connector to an API key.
Rec: host-agent-fed only — it is the whole reason tk stays a token-killer; never hold an API key.
D3Human-surface renderer engine
Vendored graph lib (Sigma/Cytoscape) vs self-built SVG/Canvas vs text/tree-first.
Rec: ship text/tree + Code Wiki first (useful immediately, no engine), add a graph engine behind the same data seam later.
D4Store: node:sqlite vs graph-DB
The graph is now richer (commit/test/doc/story kinds). Does node:sqlite + FTS still suffice, or move to an embedded graph-DB?
Rec: node:sqlite holds at repo scale; recursive CTEs cover 1-hop/BFS. Revisit only if traversal latency fails a measured budget.
D5Test → code linking
Import/call resolution (deterministic, partial) vs coverage-report ingest (precise, needs a run) vs agent-inferred intent.
Rec: deterministic import/call first, coverage optional, agent only for "what this test means".
D6Where annotations live
Same graph.db with a source tag, vs a sidecar .tk/graph.json.
Rec: graph.db for agent-fed annotations (one query core); reserve .tk/graph.json for human-authored pins / page steering.
D7How the host agent is triggered to enrich
Purely manual vs a guided tk graph enrich that tells the agent exactly what to annotate / which Code-Wiki pages to write.
Rec: provide guidance + a tk graph enrich prompt (lists un-annotated / stale high-PageRank nodes & missing pages) so enrichment is repeatable.
D8Measurement host runner
Copilot CLI headless (the stated target) vs Claude Code headless (cleaner usage) for Track-1 A/B.
Rec: Copilot CLI for fidelity; add Claude Code as a second arm if its usage stream is materially cleaner.
