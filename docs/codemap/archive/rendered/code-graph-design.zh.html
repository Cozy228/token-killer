§1b

输入层——多源数据接入

基座由多个来源融合而成，而不仅仅是代码。分界线在于哪些可以确定性提取（代码结构、git 历史、测试关联、提交→ticket 正则——tk 本地构建这些，零 LLM）与哪些需要语义映射（文档↔代码、故事↔代码意图、业务领域——这些由Agent 提供，标记为 source=host_llm）。每个节点和边都携带 source 标签和 confidence，确保二者永不混淆。

Confluence / Jira 的两种接入模式

1. 直接连接器。 tk 通过 REST（基础 URL + 配置文件/环境变量中的 token）拉取文本。拉取是确定性的，但文档↔代码映射仍需 Agent。企业级门控；自愿接入。

2. 宿主 Agent 提供（推荐）。 用户的 Agent——已经拥有 Confluence/Jira/MCP 访问权限——读取这些数据，计算文档↔代码 / 故事↔代码映射，并通过 annotate API 向 tk 写入结构化节点和边。保持 tk 无 LLM，并将困难的语义映射放在 LLM 擅长的地方。

为什么这符合"tk 不拥有任何 LLM"

tk 确定性接入解析器或 git 免费提供的所有内容。而昂贵、需要判断的部分——文档对哪段代码意味着什么、某个变更实现了哪个故事、模块服务哪个业务领域、以及代码百科的文本——正是宿主 Agent 擅长而解析器不擅长的事情。Agent（用户已经为其付费）执行这些分析并反馈结果，标记为 source=host_llm 并附带引用来源。tk 负责存储、验证锚点、跟踪过期状态——并且从不持有 API 密钥。

来源追溯是契约。确定性事实（source ∈ {static, git, test}）是可信且可复现的。Agent 断言的事实（source ∈ {host_llm, confluence, jira}）携带 confidence + 必需的引用来源，并在每个投射中明确标注——因此低置信度的文档↔代码猜测绝不会呈现为解析器证实的调用边。
§5

人类界面——代码百科 + 图谱浏览器

人类界面将共享图谱转化为人类可读、用于理解系统和协作的内容。它包含两个互补的产品——一个生成的代码百科和一个交互式图谱浏览器——都服务于同一个检索核心，在 tk 侧都是无 LLM 的（文本来自宿主 Agent，source=host_llm）。

代码百科——生成的文档

全面、层级化、仓库级别的文档——CodeWiki/RepoDoc/OpenDeepWiki 展现的能力，构建在 tk 的确定性基座之上：

  • 结构源于图谱（确定性）：基于 files/contains 的页面树；模块聚类；叶节点优先排序——类似 CodeWiki 的层次化分解。
  • 文本由宿主 Agent 提供（source=host_llm）：Agent 通过 tk_annotate 编写章节文本——类似 CodeWiki 的"订阅模式"思路，但由用户自己的 CLI Agent 执行，因此 tk 不支付任何 Token 费用。
  • 图表源于图谱 + 经过验证 + 有降级方案（RepoDoc）：基于真实边渲染的 mermaid 图、语法检查、无效时的确定性降级方案——绝不使用 LLM 幻觉。
  • 每页引用来源（OpenDeepWiki SourceFiles）：区块携带 file:line 锚点 + 新鲜度；手工编写区块永不覆盖；过期区块显示为过期。
  • 自包含 HTML 导出（CodeWiki --github-pages / GitNexus 查看器）：内联 JSON 到模板，无需 CDN，可作为 CI 产物分享。
图谱浏览器——导航

同一图谱的交互式视图——Understand-Anything 设定的 UX 标杆：

  • 概览 → 按需展开 → 聚焦：PageRank"关键文件" + 模块聚类；通过 contains/imports 按需展开子节点；聚焦 = calls 的 1 跳范围。
  • 导览模式：对 PageRank / 入口节点的有序漫游——可确定性推导，无需 LLM。
  • 详情面板：签名 + 可点击的 file:line（IDE 中的 vscode://file/...）+ 一个小的图谱派生调用片段，以及 Agent 提供的摘要/领域标签。
  • 与 tk_read 相同的锚点：人类和 Agent 指向相同的证据。
交付与成本。tk graph serve 通过 localhost HTTP 提供两个产品（可在 VS Code 的 Simple Browser 中打开）；tk graph export --html 生成自包含快照。度量：人类界面报告导航事实（显示的节点数 vs 总数、展开深度、打开的锚点数）——它不声称 Token 节省；其价值在于理解。Token 方面的故事仍然留在 Agent 界面。

索引是懒加载 + 显式的：图谱工具通过哈希刷新已变更文件，tk graph index 让团队可以预热或 CI 检查存储。Agent 界面恰好是四个常驻 MCP 工具。

tk_map

排名后的仓库地图 / 方向性回答，附带 file:line 锚点和签名片段。

tk_read

tk_search

MCP 工具的输入输出契约

每个响应都携带面向模型的输出、可解析的锚点、confidence、source 来源、过期/哈希状态以及度量事实（returnedAnchors、returnedChars）。

// tk_map —— 定向；阶梯中最便宜的一级 in { cwd, query?, token_budget? } out { answers:[{ file, line, signature, why, source, confidence }], clusters:[{ name, files[] }], next_read?, measurement } // tk_read —— 精确切片；edit_window 用于编辑场景 in { cwd, target:{ symbol | file:range | node_id }, mode:"default"|"edit_window", token_budget? } out { content, anchors:[{file,line}], content_hash, confidence, candidates?[], // 目标模糊时的窄反馈 measurement } // tk_search —— 文本 / 符号 / 调用者 / 轻量影响 in { cwd, query, mode:"text"|"symbol"|"callers"|"impact-lite", token_budget? } out { hits:[{ file, line, snippet, confidence, source }], impact?:{ callers[], co_changes_with[], owns[] }, // T1 事实，带标签 measurement } // tk_verify —— 本地 diff + 测试失败投射（从不声称正确性） in { cwd, diff?, test_output?, token_budget? } out { changed_files[], changed_symbols[], failing_tests[], anchors[], raw_pointer?, measurement }

第五个工具 tk_annotate 仅在摄取模式下暴露（见 §2b）——宿主 Agent 将语义 / 跨源标注和代码百科文本写回图谱。它不是常驻读取表面的一部分。

§2b

语义与跨源数据接入协议

这是让 tk 保持无 LLM、同时仍然承载语义/领域/业务知识、Confluence/Jira 链接和代码百科文本的关键接口。宿主 Agent（或可选的连接器）通过一个狭窄、经过验证的 API 写入——tk graph annotate（CLI）/ tk_annotate（MCP，仅摄取模式）。标注是附加性的：它们附加到现有确定性节点上，从不修改解析器事实，并且始终携带 source + confidence + 引用来源。

// tk_annotate —— 宿主 Agent 将语义/文档链接/百科文本写回 in { annotations: [ { target: { kind, qualified_name | "file:line-range" | node_id }, // 必须能解析 kind: "summary"|"domain"|"tags"|"layer" | "doc_link"|"story_link"|"test_link"|"wiki_block", value: "...text / enum / string[] / markdown...", source: "host_llm"|"confluence"|"jira", citations: [ "src/auth/login.ts:42", "confluence://SPACE/page/123", "PROJ-456" ], // 必需 confidence: "high"|"medium"|"low", content_hash: "<target 在标注时的哈希>" // 用于过期间检测 } ] } out { accepted[], rejected:[{ target, reason }], stale_now[] }
验证与安全规则
  • 每条标注必须引用一个可解析的来源（file:line、Confluence 页面、Jira 键值）；未引用 → 拒绝。
  • target 锚点必须能解析到当前图谱，否则拒绝（不允许悬空的语义）。
  • 存储时携带 source + confidence；绝不覆盖确定性事实或人类编写的区块。
  • 当目标的 content_hash 变更时自动过期——以过期状态呈现，而非静默信任。
  • 通过 (target, kind, source) 实现幂等：重新标注就地更新。
生命周期
# 1. tk 构建确定性基座（无 LLM） tk graph index # 2. 宿主 Agent 增强（使用自己的 Token，不是 tk 的） agent: 通过 tk_map/tk_read 读取图谱 agent: 映射文档/故事 ↔ 代码，推断领域， 编写代码百科文本 agent: tk_annotate({ ... source:"host_llm" ... }) # 3. 两个界面读取合并后的图谱 human: tk graph serve # 代码百科 + 浏览器 agent: tk_search / tk_map # 削减 Token # 4. 代码变更 → 标注过期 → 重新增强
§4

数据模型与静态引擎

一种类型的节点表 + 一种类型的边表（而非每种类型一张表），从第一个切片开始支持迁移。通用 nodes(kind, ..., source, confidence, content_hash, metadata_json) + edges(kind, ..., source, confidence) 模式通过新的 kind + source 值吸收所有输入源——无需新表。v1 避免 LSP（较小的支持矩阵）；ConceptNode 被移除——该角色现在由 Agent 提供的 domain/summary 标注承担。

// provenance 枚举 —— 每个节点和边携带一个 source + 一个 confidence source = "static" // tree-sitter AST —— 确定性，可信 | "git" // git log/blame/diff —— 确定性，可信 | "test" // 测试文件 AST / 覆盖率 —— 确定性结构 | "lsp" // 后续——有区分的，不会静默升级静态 | "confluence" | "jira" // 连接器拉取（文本确定性，映射由 Agent） | "host_llm" // Agent 断言的语义 —— 需要 confidence + 引用来源 confidence = "high | medium | low"
节点族 状态 说明
Repository / module / directory / file yes 来自 git 根目录 + 文件清单
Symbol / function / class / interface / type yes 来自 AST；需要精确范围 + 签名
Package / dependency / config / script partial 仅确定性清单和脚本
Commit / author yes 来自 git log/blame；source=git
Test / test_suite partial 来自测试文件 AST（+ 覆盖率）；source=test
Doc_page / wiki_space (Confluence) agent 连接器或 Agent 提供；source=confluence|host_llm
Story / epic / requirement (Jira) agent 连接器或 Agent 提供；source=jira|host_llm
WikiPage / generated block agent 代码百科：图谱派生结构 + host_llm 文本
ChangeSet yes 来自 git status/diff；新鲜度 / 影响
边族 状态 说明
contains · imports · exports · references · calls · extends · implements y/partial 静态，按语言，置信度标注
changed_by · co_changes_with · authored_by git 来自 git 历史；source=git
tests · covers test 测试 → 符号；source=test
owns T1 CODEOWNERS 启发式证据
describes · mentions · traces_to · implements_story agent 文档/故事 ↔ 代码；source=confluence|jira|host_llm
domain_of · summarizes · belongs_to_layer · documents host_llm Agent 提供的语义 + 代码百科来源追溯
投射安全契约——先结构、次候选、再精确代码、最后按差异验证
元数据→ 大纲→ 符号 / 范围→ 编辑窗口→ 完整文件 ｜ 失败投射→原始日志

编辑场景（purpose=edit）必须返回带有内容哈希 + 稳定锚点的精确切片；非精确投射被拒绝或升级。生成的 / 锁定 / 压缩 / 巨型 JSON 默认被抑制，带有明确警告 + 原始降级。强制要求保留优先的升级策略。

§12

度量框架：先度量，后构建功能

核心问题：我们如何诚实地评估 Agent 界面的搜索/读取 Token 节省——在 tk 的"已度量 ≠ 估算"分类账模型中——在构建图谱之前（ADR 0016）？（人类界面通过理解能力度量，而非 Token。）

收益 是什么 调用内可度量？ 分类
W1 输出压缩 图谱自身的响应将容器主体压缩为签名 是 measured，如同现有处理器
W2 循环避免 Agent 因为一个 tk_* 完成了任务，跳过了 5×grep + 3×read 否（反事实） 在线 opportunity / 仅离线 A/B
标题基准数字（例如"减少 47% Token"）绝大多数属于 W2。因此，框架的重心必须在轨迹级 A/B，而非调用内压缩比。tk 的 saved_tokens = raw − delivered（调用内差值）不适用于图谱查询——这就是为什么 ADR 0016 禁止图谱写入 saved_tokens。

轨道 1 — 离线 A/B 评估

已度量的证据 · 轨迹级
  • 固定任务集（11 个类别），每组每臂 N=5 次运行，取中位数（方差可达 30 倍）。
  • BASELINE = 图谱 MCP 关闭；TREATMENT = 图谱 MCP 开启；两者都保留宿主的内置 read/grep。
  • 主要指标 = uncached_input_tokens 差值，而非包含缓存的总体（缓存读取占用量 >97%）。
  • 共同度量质量：定位 F1 · AST 有效性 · FAIL_TO_PASS ∧ PASS_TO_PASS。
  • 降级回放 → omission_bug_rate 作为安全平衡指标。

轨道 2 — 在线机会核算

实时事实 · estimate_kind=opportunity · 从不求和
  • 每次查询记录机械事实：nodes_returned · files_returned · tokens_returned。
  • tool_calls_collapsed = 1（单次调用代表整个循环）。
  • reads_avoided_in_session——与轨迹分析器交叉引用。
  • 唯一显示的节省百分比是轨道 1 的参考率，标记为 measured-in-eval, not on your machine。
  • 由 Serena 独立验证，它拒绝 Token 基准测试，转而报告调用次数 / 载荷大小。

附录 A — 调研项目如何度量 Token 节省

四种度量理念。只有 A（轨迹 A/B 差值）度量 W2；B/C 回答不同的问题；D 是无实证的。

桶 A — 轨迹 A/B 差值（可模仿的模板）
codegraph 高
黄金标准；claude -p + --strict-mcp-config，7 个仓库，每臂 4 次运行，取中位数。tk 修复了其包含缓存计数的计算方式。
FastContext (MS) 高
最严谨：主 Agent vs 探测器的 Token 拆分；定位 F1。arXiv 2606.14066
SWE-ContextBench 高
Oracle 上限；缓存读取 >97%，摘要 217 vs 完整 25,634 Token。arXiv 2602.08316
SWE-Pruner 中高
动作分类 Token 化；AST 有效性 87.3% vs 0.29%。arXiv 2601.16746
claude-context 中
30 个 SWE-Bench-Verified 任务；73,373 → 44,449 Token（~39%）；方法未说明。
桶 B — 预算上限（无差值声称）
aider repo-map
二分搜索标签以适配 --map-tokens；"计数为估算值"，"从不强制执行限制"。
Probe
--max-tokens 上限；Tokenizer 未说明；无已度量的差值。
桶 C — 结构式 / 按需（未度量）
codesearch
默认返回元数据（compact=true），通过 get_chunk 获取完整代码。架构性节省；无已度量的百分比。
Serena 基准
LSP 精度标准；拒绝 Token 基准测试——轨道 2 的立场。
桶 D — 无实证的标题数字（避免）
GitNexus · Repomix · cocoindex 避免
"74%/88%"，"~70%" 但无基准 / 数据集 / 方法说明。

严谨度排名：FastContext ≈ SWE-ContextBench ≈ codegraph（高）> SWE-Pruner > claude-context > aider（上限）> Probe / codesearch > Repomix / cocoindex / GitNexus（无实证）。没有项目报告显著性检验——tk 报告中位数 + 分布范围已超过业界惯例。

§7

构建顺序——能力切片

这是产品构建顺序，而非发布阶段。每个切片以完整 + 优化状态交付，并可独立度量。绿色 = 确定性基座 / Agent 界面；蓝色 = 人类界面。

−1
先构建度量框架（ADR 0016）
A/B 运行器（图谱 MCP 开/关，无头模式）、轨迹分析器、降级回放 → omission_bug_rate。
验收条件  在任何功能上线前，在 11 任务集上获得中位数无缓存 Token 差值 + 质量 F1。
0
Node 门控 + doctor/index
Node ≥22.13 门控 + 无警告的子进程重入；tk graph doctor / index。
验收条件  核心 tk 启动不变；doctor 报告 Node/MCP/宿主/索引路径。
1
存储 + 通用模式 + 迁移
files / nodes(kind,source,confidence,content_hash,metadata_json) / edges / FTS / 度量数据。
验收条件  迁移测试；从切片 1 开始存在来源列。
2
代码 AST（TS/JS）+ tk graph map
tree-sitter 提取，稳定的节点 ID，排名。
验收条件  每个返回的锚点可解析；map 能回答方向性查询。
3
符号级增量基座
content_hash + referencer_set_hash + git-diff→changed-symbols + AST ChangeType 过滤器 + 下游 BFS。
验收条件  过期符号刷新测试；仅注释变更不触发下游重新计算。
4
Git / 提交数据源
commit/author 节点；changed_by/co_changes_with/authored_by；提交→Jira 键值正则 → traces_to。
验收条件  共同变更 + CODEOWNERS 提供 T1 影响事实。
5
Python + Java 提取 · 测试数据源
每种语言的语法；测试文件 AST → test 节点 + tests/covers 边（+ 可选覆盖率）。
验收条件  每种语言的测试夹具；测试→符号链接可解析。
6
tk graph read + tk graph search
edit_window 带哈希 + 锚点；text/symbol/callers 模式 + T1 影响，置信度标注。
验收条件  在 purpose=edit 下非精确被拒绝；低置信度命中反馈给用户。

tk_map/tk_read/tk_search/tk_verify；diff/测试失败投射。

验收条件  无引用/悬空的被拒绝；目标哈希变更时标注过期。

Confluence / Jira
首先通过 tk_annotate 由 Agent 提供；后续添加直接 REST 连接器 + Token 选择激活。
验收条件  story_link/doc_link 携带 source + 引用来源；连接器在配置后门控。

serve 接口（JSON = tk_map）→ 静态树 + 可点击 file:line → 图谱视图（概览 → 懒展开 → 聚焦）→ 详情面板。

tk install --graph + 引导 + 度量字段

§8

待决事项——需要你决定

不确定的区域，已明确列出。每个都有推荐的默认选项，以便计划可以继续推进；如有需要变更的，请指出。

D1Confluence / Jira 接入模式
直接 REST 连接器（tk 持有 token）vs 宿主 Agent 提供（Agent 读取它们，通过 tk_annotate 写入）vs 两者皆有。
建议：先 Agent 提供——保持 tk 无 LLM 和无凭据；在此接口被验证后添加连接器作为可选功能。
D2代码百科文本——谁来编写
宿主 Agent 通过 tk_annotate（订阅模式，tk 无需付费）vs 可选的 tk 侧连接到 API 密钥。
建议：仅宿主 Agent 提供——这正是 tk 保持 token-killer 身份的全部原因；绝不持有 API 密钥。
D3人类界面的渲染引擎
引入图库（Sigma/Cytoscape）vs 自建 SVG/Canvas vs 先文本/树视图。
建议：先交付文本/树 + 代码百科（立即可用，无需引擎），后续通过同一数据接口添加图引擎。
D4存储：node:sqlite vs 图谱数据库
图谱现在更加丰富（提交/测试/文档/故事类型）。node:sqlite + FTS 是否仍然足够，还是需要迁移到嵌入式图谱数据库？
建议：node:sqlite 保持不变在仓库规模下足够；递归 CTE 覆盖 1 跳/BFS。仅在遍历延迟超出已度量预算时重新考虑。
D5测试 → 代码链接
导入/调用解析（确定性，部分覆盖）vs 覆盖率报告接入（精确，需要运行）vs Agent 推断的意图。
建议：先确定性导入/调用，覆盖率可选，Agent 仅用于"这个测试意味着什么"。
D6标注存储位置
同一 graph.db 带 source 标签 vs 侧车 .tk/graph.json。
建议：Agent 提供的标注用 graph.db（一个查询核心）；将 .tk/graph.json 保留用于人类编写的固定/页面控制。
D7如何触发宿主 Agent 进行增强
纯手动 vs 引导式 tk graph enrich，它告诉 Agent 确切需要标注什么 / 编写哪些代码百科页面。
建议：提供引导 + tk graph enrich 提示（列出未标注 / 过期的 PageRank 高值节点和缺失的页面），使增强过程可重复。
D8度量宿主运行器
Copilot CLI 无头模式（声明的目标）vs Claude Code 无头模式（更干净的 usage）用于轨道 1 A/B 测试。
建议：为保真度选择 Copilot CLI；如果其用量数据流明显更清晰，则添加 Claude Code 作为第二个测试臂。
